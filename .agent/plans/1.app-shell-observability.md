# Module 1 Implementation Plan: App Shell + Observability

**Complexity:** ⚠️ **Medium** - Multiple interconnected systems (auth, database, API, frontend, observability), but each piece is straightforward.

## Overview

Module 1 establishes the application shell with:
- Supabase Auth (email/password) with Row-Level Security
- Basic chat UI with SSE streaming
- **Chat Completions API** (MiniMax/OpenAI-compatible, stateless - you manage chat history)
- LangSmith tracing on both frontend and backend

> **Note:** This uses the standard Chat Completions API instead of OpenAI's Responses API. This means:
> - You store and manage chat history yourself in the database
> - Works with any OpenAI-compatible provider (MiniMax, OpenRouter, Ollama, etc.)
> - No managed thread_id from the LLM provider

## Phase 4: Auth API

### 4.1 Schemas (`backend/app/schemas/auth.py`)
- `SignupRequest`, `LoginRequest`, `AuthResponse`, `UserResponse`

### 4.2 Auth Endpoints (`backend/app/api/auth.py`)
- `POST /api/auth/signup` - create user + profile, return tokens
- `POST /api/auth/login` - return tokens

```bash

  -d '{"email":"test@example.com","password":"testpass123"}'
```

## Phase 5: Chat Completions API

### 5.1 Chat Service (`backend/app/services/openai_service.py`)
- Uses raw OpenAI SDK with Chat Completions API (no LangChain)
- `@traceable` decorator for LangSmith
- `create_chat()` - streaming or non-streaming
- Takes `messages` array (you manage history, not the LLM)

### 5.2 Chat Schemas (`backend/app/schemas/chat.py`)
- `ConversationCreate`, `ConversationResponse`
- `MessageCreate`, `MessageResponse`
- `ChatStreamRequest`

### 5.3 Chat Endpoints (`backend/app/api/chat.py`)
- `GET /api/chat/conversations` - list user's conversations
- `POST /api/chat/conversations` - create new conversation
- `DELETE /api/chat/conversations/{id}` - delete conversation
- `GET /api/chat/conversations/{id}/messages` - list messages
- `POST /api/chat/stream` - SSE streaming endpoint

### 5.4 SSE Streaming Logic
- Build messages array from database (you manage history)
- Create conversation if needed
- Save user message to DB
- Stream from LLM, yield SSE events
- Save assistant message to DB
- Update conversation timestamp

**Validation:**
```bash
curl -X POST http://localhost:8000/api/chat/stream \
  -H "Authorization: Bearer <token>" \
  -d '{"message": "Hello"}' \
  --no-buffer
```

## MiniMax Configuration

To use MiniMax, set in `backend/.env`:
```bash
OPENAI_BASE_URL=https://api.minimax.chat/v1
OPENAI_MODEL=your-minimax-model-name
OPENAI_API_KEY=your-minimax-api-key
```

The code uses OpenAI's SDK configured for MiniMax's OpenAI-compatible endpoint, so no code changes needed for different providers.
